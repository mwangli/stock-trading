# 量化模型页面布局与展示优化方案

## 一、现状分析

### 1.1 菜单与路由

| 菜单项     | 路由         | 页面组件     | 说明                     |
|------------|--------------|--------------|--------------------------|
| 量化模型   | `/strategies`| `Strategies.tsx` | 当前为「模型卡片」页，数据全为前端 Mock |
| 策略分析   | `/analysis`  | `Analysis.tsx`   | 选股/交易策略开关与汇总，已接后端     |

「量化模型」对应的是 LSTM、情感分析等 **AI 模型** 的管理与监控，与「策略分析」（选股/交易策略开关）职责不同，路由与菜单位置合理。

### 1.2 当前 Strategies 页面问题

- **数据**：4 张卡片（LSTM、FinBERT、情感AI、规则）的标题、描述、状态、指标均为前端写死，未接后端。
- **布局**：统一 2 列卡片网格，每卡包含：图标、标题、状态 Tag、描述、3 个统计格、3 个操作按钮。信息密度高但**缺少**：
  - 真实「模型列表」（如按股票/版本的 LSTM 模型列表）；
  - 训练进度（进度条、当前 epoch、状态文案）；
  - 训练任务历史或进行中任务列表。
- **操作**：「重新训练 All」「回测」「日志」「单卡重训」均无真实逻辑或接口对接。
- **模型类型混排**：LSTM（多模型/多股票）、情感（单一服务+训练任务）、规则（无训练）混在同一层级，未在信息架构上区分「服务概览」与「模型实例列表」。

### 1.3 后端能力与缺口

| 能力           | LSTM | 情感分析 | 说明 |
|----------------|------|----------|------|
| 训练接口       | ✅ `POST /api/lstm/train` | ✅ `POST /api/models/sentiment/train` | 均有 |
| 训练状态查询   | ❌ 未暴露 | ✅ `GET /api/models/sentiment/status/{trainingId}` | LSTM 内部有 `getTrainingStatus`，需在 Controller 暴露 |
| 模型列表       | ❌ 无 | 无列表概念（单模型+健康） | LSTM 模型存 MongoDB（`LstmModelDocument`），需新增列表 API |
| 模型健康/状态  | 无统一接口 | ✅ `GET /api/models/sentiment/health` | LSTM 可考虑「当前加载模型」或「最新模型」状态 |
| 模型详情/曲线  | `ModelInfoController` 有 Mock 的 list、listTestData、listValidateData | 无 | 可改为基于真实 LSTM 元数据 + 真实或占位曲线 |

## 二、信息架构建议

### 2.1 页面定位

- **量化模型页**：以「模型类型」为一级分区，每种类型下再区分「概览 / 模型列表 / 训练与进度」。
- 建议两种模型类型优先落地：
  - **LSTM 价格预测**：多只股票、多版本模型，需要「模型列表」+「单模型训练/进度」。
  - **情感分析**：单一服务，需要「服务状态」+「训练任务」+「进度」。

### 2.2 内容分区建议

1. **顶部汇总区（可选）**  
   全局统计：已部署 LSTM 模型数、情感服务状态（已加载/未加载）、最近一次训练时间等。便于一眼看到整体健康度。

2. **LSTM 区块**  
   - **左侧或上部**：LSTM 服务概览卡片（当前加载状态、推荐指标如最新验证 Loss、训练样本数等）。  
   - **模型列表**：表格或紧凑列表，列：股票代码/名称、模型版本或创建时间、验证 Loss、状态、操作（查看曲线、重新训练、设为当前）。  
   - **训练任务**：当前进行中的训练（若有）展示进度条、当前 epoch、状态文案；支持「新建训练」入口（股票、天数、epoch 等参数）。  
   - **训练历史（可选）**：最近几次训练结果摘要（成功/失败、耗时、最终 Loss）。

3. **情感分析区块**  
   - **服务概览卡片**：模型是否已加载、最后加载时间、健康检查结果。  
   - **训练任务**：进行中任务显示进度条与状态；支持「开始训练」入口。  
   - **历史训练（可选）**：最近训练 ID、成功/失败、准确率等摘要。

4. **其他模型（FinBERT/规则）**  
   可保留为「占位卡片」或简短说明，后续再接真实数据与操作。

## 三、页面布局方案

### 3.1 整体结构（推荐）

```
+------------------------------------------------------------------+
|  [标题: 量化模型中心]  [副标题]              [全局操作: 刷新]       |
+------------------------------------------------------------------+
|  [可选] 汇总条：LSTM 模型数 | 情感服务状态 | 最近训练时间          |
+------------------------------------------------------------------+
|                                                                  |
|  +-- LSTM 价格预测 -------------------------------------------+  |
|  |  概览卡(状态/最新指标)  |  进行中训练(进度条) [新建训练]     |  |
|  +-----------------------------------------------------------+  |
|  |  模型列表 (表格)                                            |  |
|  |  代码 | 名称 | 版本/时间 | Val Loss | 状态 | 操作            |  |
|  +-----------------------------------------------------------+  |
|  |  (可选) 训练历史 或 预测曲线入口                             |  |
|  +-----------------------------------------------------------+  |
|                                                                  |
|  +-- 情感分析 -------------------------------------------------+  |
|  |  概览卡(已加载/未加载, 最后加载时间)  |  进行中训练 [开始训练]  |  |
|  +-----------------------------------------------------------+  |
|  |  (可选) 最近训练记录                                         |  |
|  +-----------------------------------------------------------+  |
|                                                                  |
|  +-- 其他模型 (FinBERT / 规则) -------------------------------+  |
|  |  简短说明或占位卡片，保持与现有风格一致                       |  |
|  +-----------------------------------------------------------+  |
|                                                                  |
+------------------------------------------------------------------+
```

- 使用 **垂直分区**：LSTM 一块、情感一块、其他一块，避免所有模型平铺成同一层级卡片。
- 每块内：**概览 + 任务/进度 + 列表** 自上而下或左概览右列表，保证「模型列表」和「训练进度」都有明确展示位。

### 3.2 展示要点

| 内容         | 展示方式建议 |
|--------------|--------------|
| 模型列表     | 表格为主：代码、名称、版本/时间、Val Loss、状态。支持分页与按股票/时间筛选（若后端支持）。 |
| 训练进度     | 进度条（0–100%）+ 状态文案（如「Epoch 3/50」「准备数据」「训练完成」）+ 可选取消按钮。 |
| LSTM 指标    | 概览卡或列表行内：验证 Loss、训练轮数、样本数；避免与「策略分析」的胜率/PnL 混淆。 |
| 情感服务状态 | 明确「已加载/未加载」+ 最后加载时间；可与健康检查接口绑定。 |
| 操作按钮     | 每个模型行：重新训练、查看曲线/日志；区块顶：新建 LSTM 训练、开始情感训练。 |

### 3.3 视觉与交互

- **状态**：在线/训练中/失败/未加载 用不同颜色 Tag 或小圆点（与现有 `#00e396` / `#ff4560` 等保持一致）。
- **进度条**：使用 Ant Design `Progress` 或自定义，与深色玻璃风格统一。
- **响应式**：大屏两列（如概览+列表），小屏上下堆叠；表格小屏可收拢为卡片列表。
- **数据兜底**：接口不可用或列表为空时，按 AGENTS 规范用 Mock 或占位文案保持页面完整、不报错。

## 四、后端接口补充建议

为实现上述布局，建议后端增加或调整如下接口（不影响现有接口）：

| 接口 | 方法 | 说明 |
|------|------|------|
| LSTM 训练状态 | `GET /api/lstm/status/{trainingId}` | 返回 progress、currentEpoch、totalEpochs、status 等，与情感侧一致。 |
| LSTM 模型列表 | `GET /api/lstm/models` 或扩展 `GET /api/modelInfo/list` | 从 MongoDB 读取 LstmModelDocument 列表（或按股票聚合），返回 code/name、modelName、createdAt、epoch、valLoss 等（valLoss 若未存可先占位）。 |
| 情感模型健康 | 已有 `GET /api/models/sentiment/health` | 前端直接使用，展示「已加载/未加载」与最后加载时间。 |

LSTM 训练为同步时，训练状态接口仍可用于「训练发起后轮询」或后续改为异步任务时的进度展示。

## 五、前端实现要点（概要）

- **状态**：LSTM 模型列表、情感健康、进行中训练 ID、训练状态等用 React state 或 Zustand 管理；训练进度可轮询 `status` 接口直到完成或失败。
- **接口层**：新增或扩展 `api/modelService.ts`（或按模块拆成 lstm.ts / sentiment.ts），封装上述 GET/POST。
- **组件拆分**：  
  - 区块级：`LstmSection`、`SentimentSection`、`OtherModelsSection`。  
  - 通用：模型列表表格式组件、训练进度条+状态文案组件、概览卡片组件。  
- **i18n**：沿用现有 `models.*` 与 `layout.strategies`，为「模型列表」「训练进度」「新建训练」等补充 key。

## 六、小结

| 项目     | 建议 |
|----------|------|
| 菜单     | 保持「量化模型」→ Strategies，「策略分析」→ Analysis。 |
| 布局     | 按模型类型分区（LSTM / 情感 / 其他）；每区：概览 + 训练与进度 + 模型列表。 |
| 展示重点 | 模型列表（表格）、训练进度（进度条+状态）、服务状态（情感已加载/未加载）。 |
| 后端     | 新增 LSTM 训练状态 GET、LSTM 模型列表 GET；情感沿用现有 train/status/health。 |
| 视觉     | 状态色统一、进度条与玻璃风格一致、空态与异常用 Mock 或占位保证页面可用。 |

按此方案可实现：**量化模型页** 清晰展示 LSTM 与情感分析模型信息、模型列表与训练进度，并与现有策略分析页在职责和视觉上区分明确。若需要，我可以再基于该方案写出具体组件结构或接口请求示例（仍不写具体实现代码）。
