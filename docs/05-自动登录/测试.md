### 2.2 Chrome 调试模式启动

Chrome 启动由后端代码自动管理（`BrowserSessionManager.startLocalChrome()`），调用 `POST /api/browser/start` 时会自动检测 Chrome 是否运行，未运行则通过 `ProcessBuilder` 自动启动：

```java
// BrowserSessionManager.java:139-160 — 自动启动逻辑
ProcessBuilder pb = new ProcessBuilder(
    "C:\\Program Files\\Google\\Chrome\\Application\\chrome.exe",
    "--remote-debugging-port=9222",
    "--user-data-dir=" + System.getProperty("user.home") + "\\chrome-sessions\\stock",
    "--disable-blink-features=AutomationControlled"
);
pb.start();
Thread.sleep(3000);  // 等待 Chrome 进程就绪
return isChromeReachable();  // 验证 CDP 端口可达
```

**自动启动触发链路**:

```
POST /api/browser/start
  → BrowserSessionManager.startBrowser()
    → isChromeReachable()          // HTTP GET localhost:9222 检测
    → 如果不可达 → tryAutoStartChrome()
      → CHROME_MODE=docker → startDockerChrome()  // docker start stock-chrome
      → 否则（默认）      → startLocalChrome()    // ProcessBuilder 启动 chrome.exe
    → 等待 3 秒后再次检测端口可达性
```

> **无需手动执行 PowerShell 脚本**，`POST /api/browser/start` 已封装全部启动逻辑。仅当自动启动失败时（如 Chrome 安装路径不同），才需要手动介入。

**手动启动（仅作为备选方案）**:

```powershell
# 当自动启动失败时，可手动执行：
taskkill /f /im chrome.exe
$chromePath = "C:\Program Files\Google\Chrome\Application\chrome.exe"
$profileDir = "$env:USERPROFILE\chrome-sessions\stock"
& $chromePath `
  --remote-debugging-port=9222 `
  --user-data-dir=$profileDir `
  --disable-blink-features=AutomationControlled
```

**验证**:

| # | 验证项 | 方法 | 预期结果 |
|---|--------|------|---------|
| E-11 | 自动启动成功 | 调用 `POST /api/browser/start`（Chrome 未运行时） | 日志输出"已启动本地 Chrome 进程"，响应 `success: true` |
| E-12 | CDP 端口可达 | 浏览器访问 `http://localhost:9222/json` | 返回 JSON 数组 |
| E-13 | 会话目录已创建 | 检查 `%USERPROFILE%\chrome-sessions\stock` | 目录存在且包含 `Default/` Profile 数据 |

#### TC-BSM-06: 会话持久化验证（Cookie / localStorage 跨重启保持）

| 项目 | 内容 |
|------|------|
| **前置条件** | Chrome 以 `--user-data-dir` 参数启动（会话目录已配置） |
| **测试步骤** | |
| 步骤 1 | 启动浏览器，调用 `POST /api/browser/start` |
| 步骤 2 | 导航到登录页，调用 `POST /api/browser/navigate/login` |
| 步骤 3 | 刷新页面，调用 `POST /api/browser/refresh`，等待 3 秒 |
| 步骤 4 | 完成一次完整登录流程，确认 Token 获取成功 |
| 步骤 5 | 记录当前 Cookie 和 localStorage 内容 |
|         | `POST /api/browser/debug/exec-js` → `return document.cookie` |
|         | `POST /api/browser/debug/exec-js` → `return JSON.stringify(localStorage)` |
| 步骤 6 | 关闭浏览器，调用 `POST /api/browser/quit` |
| 步骤 7 | 验证会话目录文件存在 |
|         | 检查 `user-data-dir` 下 `Default/Cookies`、`Default/Local Storage/` 文件 |
| 步骤 8 | 重新启动浏览器，调用 `POST /api/browser/start` |
| 步骤 9 | 导航到同一页面，调用 `POST /api/browser/navigate/login` |
| 步骤 10 | 刷新页面，调用 `POST /api/browser/refresh`，等待 3 秒 |
| 步骤 11 | 再次读取 Cookie 和 localStorage，与步骤 5 的记录对比 |
| **验证点** | |
| ① 会话目录 | `user-data-dir` 路径存在且包含 `Default/` 子目录 |
| ② Cookie 持久化 | 重启后 `document.cookie` 包含登录相关 Cookie（如 session、token） |
| ③ localStorage 持久化 | 重启后 localStorage 中的登录状态数据与关闭前一致 |
| ④ 免登录验证 | 重启后页面可能直接进入已登录状态（跳过登录页），或至少保留部分会话信息 |
| ⑤ Token 可复用 | 若会话仍有效，`GET /api/auto-login/status` 可直接返回 `loggedIn: true` |

> **说明**: 会话持久化依赖 Chrome 的 `--user-data-dir` 参数将 Cookie、localStorage、sessionStorage 保存到磁盘。此测试验证浏览器关闭并重新启动后，登录会话数据不丢失，避免每次都需重新完整登录。

```bash
# 步骤 5: 记录登录后的 Cookie
curl -X POST http://localhost:8080/api/browser/debug/exec-js \
  -H "Content-Type: application/json" \
  -d '{"script": "return document.cookie"}'

# 步骤 5: 记录登录后的 localStorage
curl -X POST http://localhost:8080/api/browser/debug/exec-js \
  -H "Content-Type: application/json" \
  -d '{"script": "return JSON.stringify(localStorage)"}'

# 步骤 6: 关闭浏览器
curl -X POST http://localhost:8080/api/browser/quit

# 步骤 7: 验证会话目录（Windows）
dir "%USERPROFILE%\chrome-sessions\stock\Default\Cookies"
dir "%USERPROFILE%\chrome-sessions\stock\Default\Local Storage"

# 步骤 8: 重新启动
curl -X POST http://localhost:8080/api/browser/start

# 步骤 9-10: 导航 + 刷新
curl -X POST http://localhost:8080/api/browser/navigate/login
sleep 3
curl -X POST http://localhost:8080/api/browser/refresh

# 步骤 11: 对比 Cookie
curl -X POST http://localhost:8080/api/browser/debug/exec-js \
  -H "Content-Type: application/json" \
  -d '{"script": "return document.cookie"}'

# 验证登录状态
curl http://localhost:8080/api/auto-login/status
```

## 十、风险与注意事项

### 10.1 测试风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 证券平台页面结构变更 | 选择器失效，登录失败 | 定期截图对比，及时更新选择器 |
| 滑块算法升级 | Sobel 距离计算不准 | 保存失败截图，分析算法偏差 |
| 邮箱验证码延迟 | 超过重试等待时间 | 增加等待间隔，考虑备选获取方式 |
| 账号被风控 | 频繁登录触发安全策略 | 控制测试频率，使用测试账号 |
| Chrome 版本更新 | CDP 协议兼容性问题 | 锁定 Chrome 版本 |

### 10.2 测试约束

- 手机验证页流程需要真实短信，依赖邮箱转发链路的可用性
- 滑块验证码每次图片不同，成功率受算法精度影响（预期 >= 70%）
- 完整回归测试需要 30-60 分钟
- 建议测试时间避开交易时段（9:30-15:00），避免影响真实交易
