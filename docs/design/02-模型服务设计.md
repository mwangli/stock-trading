- **位置**: `controller/SentimentController.java`

## 5. 模型架构

### 5.1 LSTM 模型结构

```
输入层：(batch_size, 60, 5)
    ↓
LSTM 层 1: 128 units, dropout=0.2
    ↓
LSTM 层 2: 64 units, dropout=0.2
    ↓
全连接层 1: 32 units, ReLU
    ↓
Dropout: 0.3
    ↓
全连接层 2: 1 unit, Sigmoid
    ↓
输出：涨的概率 (0-1)
```

**输入特征** (5 维):
- 开盘价 (归一化)
- 最高价 (归一化)
- 最低价 (归一化)
- 收盘价 (归一化)
- 成交量 (归一化)

### 5.2 情感分析模型

- **预训练模型**: bert-base-chinese
- **微调数据**: 金融新闻标注数据
- **输出层**: 3 分类 (利好/利空/中性)
- **映射**: 利好=1, 中性=0, 利空=-1

#### 5.2.1 资源规格说明 (Resource Specifications)

- **模型标识**: `yiyanghkust/finbert-tone-chinese`
- **磁盘占用**: 约 500MB（模型权重与 tokenizer 合并后压缩占用，解压或缓存后可能略增）。建议为模型缓存预留至少 1GB 的磁盘空间以防扩展。
- **内存 (RAM)**: 推理时建议至少 1GB 可用内存，复杂批量推理或并发时内存需求会增加，建议预留 2GB 以支持并发和 tokenizer 临时内存。
- **存储路径**:
  - MongoDB 模型仓库: 默认仍建议使用现有的 MongoDB 二进制存储（与 LSTM 模型保持一致），标识例如 `mongo:modelId`。
  - 本地缓存（可选）: `/var/lib/model-service/models/yiyanghkust-finbert-tone-chinese/`，用于减少远程加载延迟。
  - Docker 推荐挂载路径: `/opt/model-service/models/`，容器内读取路径配置在 `SentimentTrainingConfig.pretrainedModelPath`。

> 说明: 文档其他处曾使用 `利好=1, 中性=0, 利空=-1` 的映射。为避免歧义，在得分归一化部分我们将明确使用模型输出的 label 定义：0 = Neutral, 1 = Positive, 2 = Negative（与 HuggingFace 一些模型标签顺序可能不同，生产中请以模型实际输出为准并在配置中声明）。

#### 5.2.2 得分归一化 (Score Normalization)

目标: 将模型的 3 分类输出 (Negative / Neutral / Positive) 映射为统一的 0-100 分值，便于在策略模块中作为数值因子直接参与计算。

设计要点:

- 按用户建议的区间映射:
  - Negative (Label 2): 0 - 40
  - Neutral  (Label 0): 40 - 60
  - Positive (Label 1): 60 - 100
- 使用模型对预测类别的概率 (confidence / probability) 在对应区间内部线性映射。
- 基本公式示例:

```text
Score = Base + (Probability * RangeWidth)
```

具体映射:

- 若预测类别 = Negative (label = 2):
  - Base = 0, RangeWidth = 40
  - Score = 0 + P_negative * 40
- 若预测类别 = Neutral (label = 0):
  - Base = 40, RangeWidth = 20
  - Score = 40 + P_neutral * 20
- 若预测类别 = Positive (label = 1):
  - Base = 60, RangeWidth = 40
  - Score = 60 + P_positive * 40

示例: 模型预测为 Positive，概率 0.85，得分为 60 + 0.85 * 40 = 94

实现细节建议:

- 返回 DTO 中包含三个字段: 原始标签 (label)、置信度 (probability)、以及归一化后的 score (0-100)。
- 如需更平滑的跨类别连续分值，可采用加权期望映射方法: Sum_{c}(P_c * BandCenter_c)，其中 BandCenter 为每类区间中心 (Negative=20, Neutral=50, Positive=80)。该方法会将类别间不确定性自然反映到分数上，作为可选项留作后续增强。

## 8. 关键设计决策

### 8.1 模型选择

- **LSTM**: 适合时间序列预测
- **FinBERT**: 金融领域预训练，效果好
- **DJL 原生 IO**: 采用内存流加载，规避本地磁盘读写延迟
- **MongoDB 存储**: 保证模型数据的一致性与分布式访问能力

### 8.2 缓存策略

- **预测结果缓存**: 1 天有效期
- **减少重复计算**: 相同输入直接返回缓存
- **缓存失效**: 模型更新后清空缓存

### 8.3 性能优化

- **批量推理**: 支持多股票同时预测
- **模型预热**: 启动时加载模型到内存
- **异步推理**: 后台线程执行预测

## 13. 与策略模块集成 (Integration with Strategy Module)

### 13.1 接口定义

为方便策略模块（strategy-analysis）直接消费情感分析结果，建议 SentimentInference 返回的 DTO `SentimentResult` 扩展为：

```java
public class SentimentResult {
    private String text;
    private int label; // 模型原始 label: 0=Neutral, 1=Positive, 2=Negative
    private double probability; // 预测类别的置信度
    private double score; // 0-100 归一化分数
    private List<String> relatedStocks; // 识别到的股票代码
    // getters/setters
}
```

API 层 `POST /api/sentiment/analyze` 的 Response 保持兼容性，返回上述字段集合，策略模块可通过 HTTP 或内部服务调用获取批量分析结果。

### 13.2 作为选股因子的使用方式

目标: 在每日选股流程中，将情感得分作为与 LSTM 预测结果并列的数值因子参与综合评分。一个简单且可解释的融合方式如下:

- LSTM 输出: riseProbability (0-1)，映射为 0-100 的 LstmScore = riseProbability * 100
- Sentiment 输出: score (0-100)，来自上文的归一化结果
- 汇总评分计算（示例）:

```text
FinalScore = w_lstm * LstmScore + w_sentiment * SentimentScore
```

其中通常权重满足 w_lstm + w_sentiment = 1。项目当前推荐初始权重为:

- w_lstm = 0.6
- w_sentiment = 0.4

说明:

- 在候选池中对每只股票计算 FinalScore，并按降序排序进行选股。
- 若采用更复杂的多因子模型，可将情感分数作为输入特征之一，进入更高阶的加权或学习模型（例如使用简单线性回归拟合历史收益与因子权重，或使用 RankNet 类排序模型）。

### 13.3 风险与注意事项

- 模型标签与实际业务标签映射必须在部署时确认（部分预训练模型 label 顺序可能不同）。
- 当文本中包含多个股票代码时，应为每个股票分别计算情感 score，并将其与对应股票的 LSTM 结果结合。
- 对于置信度低（例如概率 < 0.55）的预测，策略模块可以选择降低情感因子的权重或标记为不确定，避免噪声影响选股结果。

### 13.4 示例流程

1. 情感服务批量分析当日新闻，返回每条新闻的 SentimentResult（含 score 和 relatedStocks）。
2. 聚合每只股票的情感分数（例如取加权平均，权重为新闻新旧或来源可信度）。
3. 从 LSTM 服务获取每只股票的 LstmScore。
4. 计算 FinalScore，按阈值或 Top-N 筛选买入候选。

> 注意: 以上接口和权重为设计建议，生产环境应基于回测结果调整权重与阈值。
