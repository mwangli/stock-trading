## 1. 模块概述

策略模块是系统的决策核心，包含两类策略：
1. **选股策略**：基于 AI 预测和情感分析的双因子选股 (天级)
2. **T+1 卖出策略**：基于分钟级价格走势的实时卖出决策 (分钟级)

### 3.2 StockSelector (股票选择器)

- **位置**: `selector/StockSelector.java`
- **职责**: 执行综合选股策略
- **输入**: 所有股票的 AI 预测结果和情感结果
- **输出**: 推荐股票列表 (Top N)
- **策略**: 双因子加权评分

**核心方法**:
```java
// 选股核心逻辑
List<StockRecommendation> selectTopN(int n);

// 获取单只股票排名
StockRanking getStockRanking(String code);

// 获取所有排名
List<StockRanking> getAllRankings();
```

**数据结构**:
```java
// 股票推荐结果
StockRecommendation {
    String stockCode;
    String stockName;
    double lstmScore;        // LSTM 预测得分
    double sentimentScore;   // 情感分析得分
    double totalScore;       // 综合得分
    int rank;                // 排名
}

// 评分股票
ScoredStock {
    String code;
    double score;
}
```

## 4. 选股策略设计

### 4.1 双因子模型

```
LSTM 预测结果 + 情感分析结果
    ↓
因子计算器 (标准化)
    ↓
综合得分 = LSTM×0.6 + 情感×0.4
    ↓
股票排名 (降序排序)
    ↓
Top N 筛选 (默认 10)
    ↓
输出：推荐股票列表
```

### 4.2 因子权重

| 因子 | 权重 | 说明 |
|------|------|------|
| LSTM 预测 | 60% | 价格涨跌概率 |
| 情感分析 | 40% | 新闻情感得分 |

### 4.3 选股参数

| 参数名 | 默认值 | 说明 |
|--------|--------|------|
| lstmWeight | 0.6 | LSTM 因子权重 |
| sentimentWeight | 0.4 | 情感因子权重 |
| topN | 10 | 选股数量 |
| minScore | 0.5 | 最低综合得分 |

### 5.2 卖出指标

#### 移动止损 (40% 权重)
- **触发条件**: `当前价 <= 最高价 × (1 - 回撤容忍度)`
- **参数**:
  - 基础回撤容忍度：2%
  - 高波动股票阈值：3% (日振幅>5%)
  - 低波动股票阈值：1% (日振幅<3%)

#### RSI 超买 (20% 权重)
- **触发条件**: `RSI(14) > 75 AND 当前价 > 开盘价 × 1.02`
- **参数**:
  - RSI 周期：14 分钟
  - 超买阈值：75
  - 最小涨幅要求：2%

#### 成交量背离 (20% 权重)
- **触发条件**: `当前价是最近5分钟最高 AND 最近1分钟成交量 < 前5分钟均量 × 0.8`
- **参数**:
  - 价格创新高窗口：5 分钟
  - 成交量比较窗口：5 分钟
  - 萎缩阈值：80%

#### 布林带突破 (20% 权重)
- **触发条件**: `当前价 > 上轨 × 1.005 AND 连续3分钟在上轨之上`
- **参数**:
  - 周期：20 分钟
  - 标准差倍数：2
  - 连续突破要求：3 根 K 线

### 5.3 动态阈值

| 当日涨幅 | 卖出阈值 | 逻辑 |
|----------|----------|------|
| >3% | 50% | 已有不错收益，可以更积极 |
| 1% - 3% | 60% | 正常情况 |
| <1% | 80% | 收益有限，需更强信号 |

## 9. 关键设计决策

### 9.1 双策略并行

- **选股策略**: 天级，基于 AI 预测，决定买什么
- **T+1 策略**: 分钟级，基于价格走势，决定何时卖
- **原因**: 买入和卖出决策逻辑不同，需要独立策略

### 9.2 多指标聚合

- **设计**: 4 个指标独立计算，加权聚合
- **优势**: 
  - 避免单一指标误判
  - 各指标互补，提高准确性
  - 权重可调整，适应不同市场

### 9.3 动态阈值

- **设计**: 根据当日收益自动调整卖出阈值
- **原因**: 
  - 收益高时可更积极锁定利润
  - 收益低时需更强信号避免频繁交易

## 10. 与其他模块的关系

- **data-collector**: 提供股票列表和分钟级价格数据
- **model-service**: 提供 LSTM 预测和情感分析结果 (用于选股策略)
- **trading-executor**: 提供交易信号、持仓信息和卖出决策执行

## 15. 模块间通信设计

### 15.1 通信方案选型

经过对比分析，选择 **Redis Pub/Sub** 作为模块间通信方案：

| 方案 | 实时性 | 解耦程度 | 复杂度 | 独立启动 | 选择 |
|------|--------|--------|--------|----------|------|
| Spring Event | ⭐⭐⭐⭐⭐ | 低 | ⭐ | ❌ | - |
| Redis Pub/Sub | ⭐⭐⭐⭐⭐ | 高 | ⭐⭐ | ✅ | ✅ |
| Redis Stream | ⭐⭐⭐⭐ | 高 | ⭐⭐⭐ | ✅ | - |

**选择理由**：
- ✅ 交易信号实时性要求极高，需要毫秒级响应
- ✅ 消费者离线时消息丢失可接受（过时信号无价值）
- ✅ 支持聚合启动和独立启动
- ✅ 实现简单，无额外依赖（Redis 已有）

### 15.2 通信架构

```
┌─────────────────┐    PUBLISH    ┌──────────────┐    SUBSCRIBE    ┌─────────────────┐
│ strategy-analysis│ ───────────→ │    Redis     │ ←─────────── │ trading-executor│
│  SignalPublisher │   trading:    │   (实时转发)  │                 │  SignalListener │
│                  │   signal      │              │                 │                 │
└─────────────────┘              └──────────────┘                 └─────────────────┘
        │                                                               │
        │ "立即发布"                                                     │ "立即执行"
        │                                                               │
        └───────────────────────────────────────────────────────────────┘
                          实时性 ✓ 解耦 ✓ 轻量 ✓
```

### 15.3 SignalPublisher (信号发布器)

- **位置**: `publisher/SignalPublisher.java`
- **职责**: 将交易信号发布到 Redis 频道
- **功能**:
  - 序列化信号为 JSON
  - 发布到 `trading:signal` 频道
  - 记录发布日志

**核心方法**:
```java
@Service
@RequiredArgsConstructor
public class SignalPublisher {
    private final StringRedisTemplate redisTemplate;
    
    private static final String CHANNEL = "trading:signal";
    
    // 发布交易信号
    public void publishSignal(TradingSignalDto signal) {
        String json = toJson(signal);
        redisTemplate.convertAndSend(CHANNEL, json);
        log.info("发布交易信号: {} {}", signal.getSignalType(), signal.getStockCode());
    }
    
    // 批量发布
    public void publishSignals(List<TradingSignalDto> signals) {
        for (TradingSignalDto signal : signals) {
            publishSignal(signal);
        }
    }
}
```

| 特性 | 说明 |
|------|------|

| 启动模式 | 支持聚合启动、独立启动 |

### 15.7 设计决策

#### 为什么选择 Pub/Sub 而不是 Stream？

**业务场景特点**：
- 交易信号是**实时性要求极强**的数据
- 如果消费者不能立即响应，后续恢复的数据**没有意义**
- 过时信号 = 无价值 → 丢弃合理

**Pub/Sub 与 Stream 对比**:
| 维度 | Redis Pub/Sub | Redis Stream |
|------|---------------|---------------|
| **实时性** | ⭐⭐⭐⭐⭐ 推送，毫秒级 | ⭐⭐⭐⭐ 轮询/阻塞，稍慢 |
| **消息存储** | ❌ 不存储 | ✅ 存储（但无意义） |
| **离线消费** | ❌ 不支持（但不需要） | ✅ 支持（但无意义） |
| **实现复杂度** | ⭐ 极简 | ⭐⭐⭐ 中等 |
| **消息确认** | ❌ 无（但不需要） | ✅ 有（但不需要） |

**结论**：Redis Stream 的优势（持久化、离线消费、消息确认）在交易信号场景下**完全用不上**，反而增加了复杂度。
