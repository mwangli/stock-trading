# 04-执行模块 - 设计文档

## 模块概述

执行模块（`com.stock.executor`）是系统的交易执行层，负责风控检查、订单执行和持仓管理，确保交易安全可控。

## 核心职责

1. **风控检查** - 交易前的风险评估和规则检查
2. **订单执行** - 买入/卖出订单的生成和执行
3. **持仓管理** - 持仓跟踪、盈亏统计
4. **账户管理** - 账户资金、可用余额管理
5. **交易记录** - 交易历史、委托记录管理

## 核心组件

### 1. RiskController (风控检查器)

- **职责**: 交易前的风险评估
- **检查项**:
  - 止损检查：是否触发止损线
  - 仓位检查：是否超过仓位限制
  - 熔断检查：是否触发熔断机制
- **输出**: RiskCheckResult(通过/拒绝，原因)

### 2. TradeExecutor (交易执行器)

- **职责**: 执行买入/卖出交易
- **功能**:
  - 生成交易订单
  - 执行交易 (模拟/实盘)
  - 确认交易结果
  - 更新持仓和账户
- **模式**: 支持模拟交易和实盘交易切换

### 3. OrderManager (订单管理器)

- **职责**: 管理交易订单的完整生命周期
- **功能**:
  - 订单创建和存储
  - 订单状态跟踪
  - 订单修改和撤销
  - 订单查询和统计

### 4. PositionManager (持仓管理器)

- **职责**: 管理股票持仓
- **功能**:
  - 持仓增减更新
  - 持仓成本计算
  - 浮动盈亏统计
  - 持仓分布分析

### 5. AccountManager (账户管理器)

- **职责**: 管理交易账户资金
- **功能**:
  - 可用余额管理
  - 资金冻结/解冻
  - 资金流水记录
  - 账户统计分析

## 风控规则

### 止损规则

| 规则名 | 阈值 | 说明 |
|--------|------|------|
| 日亏损限制 | 3% | 当日累计亏损超过 3% 禁止买入 |
| 月亏损限制 | 10% | 当月累计亏损超过 10% 禁止买入 |
| 个股止损 | 8% | 单只股票亏损超过 8% 触发卖出 |
| 总止损 | 15% | 总账户亏损超过 15% 清仓 |

### 仓位规则

| 规则名 | 阈值 | 说明 |
|--------|------|------|
| 单股仓位上限 | 30% | 单只股票市值不超过总资产 30% |
| 总仓位上限 | 80% | 股票总市值不超过总资产 80% |
| 最低仓位 | 20% | 保持至少 20% 现金应对风险 |
| 行业集中度 | 50% | 同一行业股票不超过 50% |

### 交易限制

| 规则名 | 限制 | 说明 |
|--------|------|------|
| 单笔最小金额 | 1000 元 | 低于此金额不执行 |
| 单日交易次数 | 10 次 | 超过次数禁止交易 |
| T+1 规则 | 强制 | 当日买入次日才能卖出 |
| 涨跌停限制 | 强制 | 涨停不买入，跌停不卖出 |

## 数据流程

### 买入交易流程

```
交易信号 (买入)
    ↓
风控检查 (止损/仓位/熔断)
    ↓
通过？──否──→ 拒绝并记录原因
    │
    是
    ↓
生成买入订单
    ↓
执行交易 (模拟/实盘)
    ↓
更新持仓 (增加)
    ↓
更新账户 (扣减资金)
    ↓
记录交易流水
    ↓
输出：交易结果
```

```

    ↓

    ↓
通过？──否──→ 拒绝并记录原因
    │
    是
    ↓

    ↓
执行交易 (模拟/实盘)
    ↓
更新持仓 (减少)
    ↓
更新账户 (增加资金)
    ↓
计算盈亏
    ↓
记录交易流水
    ↓
输出：交易结果
```

## 技术栈

- **框架**: Spring Boot 3.2.2

- **事务**: Spring @Transactional

## 关键设计决策

### 1. 风控优先

- **设计**: 所有交易必须先通过风控检查
- **实现**: 风控检查作为交易执行的前置条件
- **原因**: 控制风险是交易的第一原则

### 2. 模拟/实盘分离

- **设计**: 支持模拟交易和实盘交易两种模式
- **切换**: 通过配置开关切换
- **原因**: 
  - 模拟模式用于测试策略
  - 实盘模式用于真实交易
  - 代码复用，降低风险

### 3. T+1 强制检查

- **设计**: 系统强制检查 T+1 规则
- **实现**: 记录每笔买入日期，卖出时检查
- **原因**: A 股交易规则，不可违反

### 4. 交易原子性

- **设计**: 交易执行使用数据库事务
- **实现**: @Transactional 注解保证原子性
- **原因**: 确保订单、持仓、账户数据一致性

### 5. 多级风控

- **设计**: 多层次风控检查
- **层次**: 
  - 账户级 (总亏损、总仓位)
  - 股票级 (单股亏损、单股仓位)
  - 交易级 (单笔金额、交易频率)
- **原因**: 全方位控制风险

## 与其他模块的关系

- **数据采集模块**: 获取实时价格用于盈亏计算
- **AI 模型模块**: 不直接交互
- **策略模块**: 接收交易信号并执行

## 交易模式

### 模拟交易模式

- **特点**: 不真实下单，只模拟成交
- **用途**: 策略测试、系统验证
- **配置**: `trade.mode=simulated`

### 实盘交易模式

- **特点**: 真实下单到券商
- **用途**: 真实交易
- **配置**: `trade.mode=live`
- **要求**: 需要券商 API 接入
