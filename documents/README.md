## 4 个核心模块

| 模块 | 包路径 | 职责 |
|------|--------|------|
| 数据采集 | com.stock.dataCollector | 股票数据、新闻采集 |
| AI 模型 | com.stock.modelService | 价格预测、情感分析 |
| 交易策略 | com.stock.strategyAnalysis | 选股决策、信号生成 |
| 交易执行 | com.stock.tradingExecutor | 风控检查、订单执行 |

## 3 种文档类型

### 需求文档 (requirements/)

定义每个模块的功能需求、性能需求、接口需求等。

- **用途**: 开发前明确需求，开发后验收依据
- **内容**: 功能需求、性能指标、验收标准
- **优先级**: P0(必须)、P1(重要)、P2(可选)

### 设计文档 (design/)

描述每个模块的系统设计、核心组件、数据流程等。

- **用途**: 指导开发实现，便于后续维护
- **内容**: 架构图、组件说明、接口定义、技术栈
- **重点**: 关键设计决策和原因

### 测试文档 (test/)

提供每个模块的测试用例、测试方法、验收标准等。

- **用途**: 指导测试执行，确保质量
- **内容**: 单元测试、集成测试、系统测试、性能测试
- **指标**: 测试覆盖率、通过率、性能指标

## 开发流程

```
需求分析 → 设计评审 → 代码实现 → 测试验证 → 代码审查 → 提交部署
    ↓          ↓          ↓          ↓          ↓          ↓
更新需求文档  更新设计文档  编写代码    执行测试    代码审查    Git 提交
```

## 快速开始

### 新成员入门

1. 先阅读 [00-系统整体需求](./requirements/00-系统整体需求.md) 了解系统全貌
2. 阅读 [00-系统架构设计](./design/00-系统架构设计.md) 了解技术架构
3. 根据需要查阅各模块详细需求和设计
4. 开发前阅读对应的测试文档了解验收标准

### 开发新功能

1. 更新或创建需求文档
2. 更新或创建设计文档
3. 编写代码
4. 执行测试文档中的测试用例
5. 更新相关文档

### 修复 Bug

1. 定位问题模块
2. 查阅对应设计文档了解实现
3. 修复代码
4. 执行对应测试文档的测试用例
5. 更新文档 (如有必要)

## 文档维护

### 文档更新原则

- **需求变更**: 先更新需求文档，再修改代码
- **设计变更**: 先更新设计文档，再修改实现
- **测试用例**: 随代码变更同步更新
- **版本控制**: 文档纳入 Git 版本管理

### 文档命名规范

- 文件名：`序号 - 模块名 - 文档类型.md`
- 序号：00(整体), 01(数据采集), 02(AI 模型), 03(交易策略), 04(交易执行)
- 文档类型：需求/设计/测试

### 文档模板

各类型文档应包含以下基本结构：

**需求文档:**
- 模块概述
- 功能需求 (优先级、验收标准)
- 性能需求
- 接口需求
- 依赖关系

**设计文档:**
- 模块概述
- 系统架构
- 核心组件
- 数据流程
- 技术栈
- 关键设计决策

**测试文档:**
- 测试策略
- 单元测试
- 集成测试
- 系统测试
- 性能测试
- 测试用例清单
