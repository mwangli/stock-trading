## 2. 模块划分

### 2.1 Maven 多模块结构

后端采用 Maven 多模块项目结构，每个模块独立子项目：

```
backend/
├── pom.xml                    # 父 POM (依赖管理)
├── data-collector/            # 数据采集模块
│   ├── pom.xml
│   └── src/main/java/com/stock/dataCollector/
├── model-service/             # AI 模型模块
│   ├── pom.xml
│   └── src/main/java/com/stock/modelService/
├── strategy-analysis/         # 策略分析模块
│   ├── pom.xml
│   └── src/main/java/com/stock/strategyAnalysis/
└── trading-executor/          # 交易执行模块 (主启动类)
    ├── pom.xml
    └── src/main/java/com/stock/tradingExecutor/
```

### 2.2 模块职责

| 模块 | Maven ArtifactId | 包路径 | 职责 | 关键技术 |
|------|------------------|--------|------|----------|
| 数据采集 | data-collector | com.stock.dataCollector | 股票数据、新闻采集 | MongoDB, Spring Scheduler |
| AI 模型 | model-service | com.stock.modelService | 价格预测、情感分析 | DJL, LSTM, PyTorch |
| 策略模块 | strategy-analysis | com.stock.strategyAnalysis | 选股决策、信号生成 | 双因子模型 |
| 执行模块 | trading-executor | com.stock.tradingExecutor | 风控检查、交易执行 | Spring Boot 主启动 |

### 2.3 模块依赖关系

```
trading-executor (主模块)
    ├── depends on: strategy-analysis
    ├── depends on: model-service
    └── depends on: data-collector

strategy-analysis
    ├── depends on: model-service
    └── depends on: data-collector

model-service
    └── depends on: data-collector

data-collector
    └── (基础模块，无内部依赖)
```

## 3. 核心业务流程

### 3.1 数据采集流程

```
证券平台 API
    ↓
SecuritiesClient (API 客户端)
    ↓
StockDataService (数据服务)
    ↓
数据清洗和验证
    ↓
存储 (MongoDB)
    ↓
供 AI 模型模块使用
```

### 3.2 AI 预测流程

```
K 线数据 (MongoDB) → LSTM 训练/推理 → 涨跌概率
新闻数据 (MySQL) → 情感分析 → 情感得分
    ↓
输出到策略模块
```

### 3.3 策略决策流程

```
LSTM 预测结果 + 情感分析结果
    ↓
因子计算器 (标准化)
    ↓
综合得分 = LSTM×0.6 + 情感×0.4
    ↓
股票排名 (降序排序)
    ↓
决策引擎 (生成交易信号)
    ↓
输出到执行模块
```

### 3.4 交易执行流程

```
交易信号
    ↓
风控检查 (止损/仓位/熔断)
    ↓
通过？─否─→ 拒绝
    │
    是
    ↓
执行交易 (模拟/实盘)
    ↓
更新持仓和账户
    ↓
记录交易流水
```

## 7. 关键设计决策

### 7.1 多模块架构

- **选择**: Maven 多模块项目结构
- **原因**:
  - 各模块职责清晰，独立开发
  - 模块间依赖明确，便于管理
  - 支持独立测试和部署
  - 代码复用性高

### 7.2 数据库选型

- MySQL: 结构化业务数据
- MongoDB: 时序数据 (K 线)
- Redis: 缓存和会话

### 7.3 模块化设计

- 各模块职责清晰
- 模块间松耦合
- 便于独立测试和维护
