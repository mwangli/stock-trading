## 2. 部署方案对比

### 2.1 原生部署 vs Docker部署

#### 方案A：原生部署（Traditional Deployment）

**架构图：**
```
┌─────────────────────────────────────────────────────────┐
│                    服务器 (2C4G)                        │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐    │
│  │ Nginx   │  │ Java    │  │ Python  │  │ MySQL   │    │
│  │ :80     │  │ :8080   │  │ :8001   │  │ :3306   │    │
│  └─────────┘  └─────────┘  └─────────┘  └─────────┘    │
│  ┌─────────┐  ┌─────────┐                              │
│  │ MongoDB │  │ Redis   │                              │
│  │ :27017  │  │ :6379   │                              │
│  └─────────┘  └─────────┘                              │
└─────────────────────────────────────────────────────────┘
```

**优点：**
1. **性能最优** - 无虚拟化开销，直接利用硬件资源
2. **资源占用低** - 相比Docker额外占用更少内存
3. **调试方便** - 直接查看进程日志，排查问题直观
4. **无额外学习成本** - 熟悉传统Linux服务管理

**缺点：**
1. **环境一致性差** - 不同服务器环境可能导致"在我机器上能运行"
2. **依赖冲突** - 不同项目可能需要不同版本的JDK、Python
3. **部署复杂** - 需要手动安装所有依赖，步骤繁琐
4. **扩缩容困难** - 难以快速迁移或扩展
5. **回滚困难** - 版本回退需要重新部署

**资源估算：**
| 服务 | 内存占用 | CPU占用 |
|------|----------|---------|
| Nginx | 50MB | 5% |
| Java (Spring Boot) | 300-512MB | 20% |
| Python (FastAPI) | 200-300MB | 15% |
| MySQL | 256MB | 15% |
| MongoDB | 128MB | 10% |
| Redis | 64MB | 5% |
| **总计** | **约1.0-1.3GB** | **约70%** |

#### 方案B：Docker部署（Container Deployment）

**架构图：**
```
┌─────────────────────────────────────────────────────────┐
│                    服务器 (2C4G)                        │
│  ┌─────────────────────────────────────────────────┐   │
│  │              Docker Engine                      │   │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐           │   │
│  │  │Frontend │ │ Backend │ │  AI Svc │           │   │
│  │  │ Container│ │Container│ │Container│           │   │
│  │  └─────────┘ └─────────┘ └─────────┘           │   │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐           │   │
│  │  │  MySQL  │ │ MongoDB │ │  Redis  │           │   │
│  │  │Container│ │Container│ │Container│           │   │
│  │  └─────────┘ └─────────┘ └─────────┘           │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

**优点：**
1. **环境一致** - 镜像构建一次，到处运行，确保一致性
2. **快速部署** - 拉取镜像即可启动，部署时间短
3. **易于扩缩容** - 可以快速启动多个实例
4. **资源隔离** - 各服务独立，不会相互影响
5. **回滚简单** - 指定镜像版本即可回退
6. **维护简单** - 统一使用docker-compose管理

**缺点：**
1. **资源开销** - Docker引擎占用额外内存约100-200MB
2. **性能损耗** - 网络IO有轻微虚拟化开销（约3-5%）
3. **学习成本** - 需要了解Docker和docker-compose
4. **调试复杂** - 需要进入容器内部查看日志
5. **磁盘占用** - 镜像和容器会占用额外磁盘空间

**资源估算：**
| 服务 | 内存占用 | CPU占用 |
|------|----------|---------|
| Docker Engine | 100-200MB | 5% |
| Frontend (Nginx) | 64MB | 5% |
| Backend (Java) | 300-512MB | 20% |
| AI Service (Python) | 200-400MB | 15% |
| MySQL | 256MB | 15% |
| MongoDB | 128MB | 10% |
| Redis | 64MB | 5% |
| **总计** | **约1.1-1.6GB** | **约75%** |

### 2.2 前后端分离部署 vs 整合部署

#### 方案A：前后端分离部署

**架构图：**
```
┌─────────────────────────────────────────────────────────┐
│                      Nginx (80)                        │
│  ┌─────────────────────┬───────────────────────────┐   │
│  │  /api/* → Backend   │  / (SPA) → Frontend       │   │
│  │  /ai-api/* → Python │  静态资源                  │   │
│  └─────────────────────┴───────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
           │                          │
           ▼                          ▼
┌─────────────────────┐    ┌─────────────────────────────┐
│  Backend :8080      │    │  Frontend (构建产物)         │
│  - API接口          │    │  - HTML/CSS/JS              │
│  - 业务逻辑         │    │  - 独立部署                  │
└─────────────────────┘    └─────────────────────────────┘
```

**优点：**
1. **独立开发** - 前端和后端可以独立迭代，互不影响
2. **独立部署** - 前端修改无需重启后端
3. **CDN加速** - 前端静态资源可以部署到CDN
4. **更好的扩展性** - 可以单独扩展前端或后端
5. **技术栈灵活** - 前后端可以使用不同技术栈

**缺点：**
1. **跨域问题** - 需要处理CORS或使用Nginx代理
2. **部署复杂** - 需要管理多个部署单元
3. **调试不便** - 前后端联调需要更多协调

**推荐配置：**
```nginx
server {
    listen 80;
    server_name _;

    # 前端静态资源
    location / {
        root /usr/share/nginx/html;
        index index.html;
        try_files $uri $uri/ /index.html;
    }

    # 后端API代理
    location /api/ {
        proxy_pass http://backend:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # AI服务代理
    location /ai-api/ {
        proxy_pass http://stock-service:8001;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

#### 方案B：前后端整合部署

**架构图：**
```
┌─────────────────────────────────────────────────────────┐
│                    Spring Boot                         │
│  ┌─────────────────────────────────────────────────┐   │
│  │  Backend :8080                                  │   │
│  │  ├── /api/* (业务API)                           │   │
│  │  ├── /static/* (前端构建产物)                    │   │
│  │  └── /templates/* (模板文件)                     │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

**优点：**
1. **简化部署** - 只需要部署一个包
2. **无跨域问题** - 前后端同源
3. **调试简单** - 本地开发更方便

**缺点：**
1. **不适用于React SPA** - React使用客户端路由，需要Nginx处理
2. **后端压力大** - 需要处理静态文件请求
3. **不灵活** - 前后端紧耦合

**适用场景：**
- JSP/Thymeleaf等服务端渲染框架
- 小型项目
- 团队规模小

## 4. 推荐方案

### 4.1 最终推荐：Docker Compose 前后端分离部署

**推荐理由：**

1. **环境一致性保障**
   - Docker镜像确保开发、测试、生产环境一致
   - 避免"在我机器上能运行"的问题

2. **资源利用合理**
   - 2C4G资源可以满足所有服务运行
   - Docker资源限制防止单一服务耗尽资源

3. **维护成本低**
   - docker-compose统一管理，一行命令启动/停止
   - 版本管理和回滚简单

4. **扩展性强**
   - 未来升级到Kubernetes平滑
   - 服务可独立扩展

5. **项目已有支持**
   - 仓库已包含docker-compose.yml
   - 各服务已有Dockerfile

### 4.2 推荐的docker-compose配置

```yaml
version: '3.8'

services:
  # 前端服务
  frontend:
    build:
      context: ./stock-frontend
      dockerfile: Dockerfile
    container_name: stock-frontend
    ports:
      - "8000:80"
    depends_on:
      - backend
    networks:
      - stock-network
    restart: unless-stopped
    deploy:
      resources:
        limits:
          cpus: '0.25'
          memory: 128M

  # 后端服务
  backend:
    build:
      context: ./stock-backend
      dockerfile: Dockerfile
    container_name: stock-backend
    ports:
      - "8080:8080"
    environment:
      - SPRING_PROFILES_ACTIVE=prod
      - SPRING_DATASOURCE_URL=jdbc:mysql://mysql:3306/stock_trading
      - SPRING_DATA_MONGODB_URI=mongodb://mongo:27017/stock_trading
      - SPRING_REDIS_HOST=redis
      - SPRING_REDIS_PORT=6379
      - STOCK_AI_SERVICE_URL=http://stock-service:8001
    depends_on:
      - mysql
      - mongo
      - redis
      - stock-service
    networks:
      - stock-network
    restart: unless-stopped
    deploy:
      resources:
        limits:
          cpus: '0.5'
          memory: 512M

  # Python AI服务
  stock-service:
    build:
      context: ./stock-service
      dockerfile: Dockerfile
    container_name: stock-ai-service
    ports:
      - "8001:8001"
    environment:
      - PYTHONUNBUFFERED=1
      - DEBUG=false
    volumes:
      - stock-service-models:/app/models
    networks:
      - stock-network
    restart: unless-stopped
    deploy:
      resources:
        limits:
          cpus: '0.5'
          memory: 2G
        reservations:
          memory: 512M

  # MySQL数据库
  mysql:
    image: mysql:8.0
    container_name: stock-mysql
    environment:
      - MYSQL_ROOT_PASSWORD=root123
      - MYSQL_DATABASE=stock_trading
    ports:
      - "13306:3306"
    volumes:
      - mysql-data:/var/lib/mysql
    networks:
      - stock-network
    restart: unless-stopped
    deploy:
      resources:
        limits:
          cpus: '0.5'
          memory: 512M

  # MongoDB
  mongo:
    image: mongo:6.0
    container_name: stock-mongo
    environment:
      - MONGO_INITDB_DATABASE=stock_trading
    ports:
      - "27017:27017"
    volumes:
      - mongo-data:/data/db
    networks:
      - stock-network
    restart: unless-stopped
    deploy:
      resources:
        limits:
          cpus: '0.25'
          memory: 256M

  # Redis缓存
  redis:
    image: redis:7-alpine
    container_name: stock-redis
    ports:
      - "6379:6379"
    volumes:
      - redis-data:/data
    networks:
      - stock-network
    restart: unless-stopped
    deploy:
      resources:
        limits:
          cpus: '0.25'
          memory: 256M

networks:
  stock-network:
    driver: bridge

volumes:
  mysql-data:
  mongo-data:
  redis-data:
  stock-service-models:
```

## 5. 实施步骤

### 5.1 Docker部署实施步骤

#### 步骤1：服务器环境准备

```bash
# 1. 更新系统
yum update -y

# 2. 安装Docker
curl -fsSL https://get.docker.com | sh
systemctl start docker
systemctl enable docker

# 3. 安装Docker Compose
curl -L "https://github.com/docker/compose/releases/download/v2.24.0/docker-compose-$(uname -s)-$(uname -m)" -o /usr/local/bin/docker-compose
chmod +x /usr/local/bin/docker-compose

# 4. 配置Docker镜像加速（可选）
mkdir -p /etc/docker
cat > /etc/docker/daemon.json <<EOF
{
  "registry-mirrors": ["https://docker.mirrors.ustc.edu.cn"]
}
EOF
systemctl daemon-reload
systemctl restart docker
```

#### 步骤2：部署配置

```bash
# 1. 创建项目目录
mkdir -p /opt/stock-trading
cd /opt/stock-trading

# 2. 上传项目文件（或git clone）
git clone <your-repo-url> .

# 3. 配置环境变量（如需要）
cp .env.example .env
# 编辑 .env 文件配置敏感信息
```

#### 步骤3：构建和启动

```bash
# 1. 构建并启动所有服务
docker-compose up -d --build

# 2. 查看服务状态
docker-compose ps

# 3. 查看日志
docker-compose logs -f

# 4. 验证服务
curl http://localhost:8000
curl http://localhost:8080/api/health
curl http://localhost:8001/api/health
```

#### 步骤4：服务管理

```bash
# 停止服务
docker-compose down

# 重启特定服务
docker-compose restart backend

# 查看资源使用
docker stats

# 清理未使用的资源
docker system prune
```

### 5.2 生产环境注意事项

1. **数据持久化**
   - 确保volume正确挂载
   - 定期备份数据卷

2. **安全配置**
   - 修改默认密码
   - 配置防火墙规则
   - 启用Docker资源限制

3. **监控告警**
   - 配置日志收集
   - 设置资源告警阈值

4. **自动启动**
   - 配置docker-compose服务开机自启
   - 添加健康检查

## 6. 风险与注意事项

### 6.1 潜在风险

| 风险 | 等级 | 应对措施 |
|------|------|----------|
| 内存不足 | 高 | 监控内存使用，设置告警 |
| Docker存储占满 | 中 | 定期清理未使用镜像和容器 |
| 服务启动失败 | 中 | 配置健康检查和自动重启 |
| 数据丢失 | 高 | 定期备份 volume 数据 |

### 6.2 监控建议

```bash
# 监控容器资源使用
docker stats

# 监控磁盘使用
df -h

# 监控内存使用
free -h

# 监控服务日志
docker-compose logs -f --tail=100
```

### 6.3 备份策略

```bash
# 备份MySQL数据
docker exec stock-mysql mysqldump -u root -p stock_trading > backup.sql

# 备份Redis数据
docker exec stock-redis redis-cli SAVE

# 备份MongoDB数据
docker exec stock-mongo mongodump --out=/backup
```

## 7. 总结

**推荐采用Docker Compose前后端分离部署方案**，理由如下：

1. ✅ 环境一致性 - 解决跨环境问题
2. ✅ 资源可控 - Docker资源限制保护系统
3. ✅ 维护简单 - 统一管理命令
4. ✅ 易于扩展 - 平滑升级K8s
5. ✅ 项目已有支持 - 降低实施难度

在2C4G服务器资源下，该方案可以稳定运行，并预留了足够的扩展空间。
