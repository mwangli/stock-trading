# AI模型模块 - 智能服务设计

## 模块说明

AI模型模块负责AI训练、推理和模型管理，基于DJL框架实现纯Java深度学习。

## 包含功能

| 序号 | 功能 | 说明 |
|------|------|------|
| 1 | 情感分析 | FinBERT金融文本分类 |
| 2 | LSTM预测 | DJL LSTM时序预测 |
| 3 | 实体识别 | 多策略股票实体匹配 |
| 4 | 模型训练 | 纯Java端到端训练 |
| 5 | 模型管理 | 版本管理、加载、切换 |

## 核心类

| 类名 | 说明 |
|------|------|
| StockLSTM | LSTM网络定义(DJL Block) |
| SentimentInference | FinBERT情感分析推理服务 |
| LstmInference | LSTM推理服务 |
| EntityRecognizer | 股票实体识别器 |
| NewsProcessor | 新闻处理流水线 |
| SentimentAggregator | 情感聚合服务 |
| LstmTrainer | LSTM训练服务 |
| SentimentTrainer | 情感分析训练服务 |
| DjlModelManager | DJL模型管理器 |

## 技术栈

- DJL 0.36.0
- PyTorch Engine
- FinBERT (中文金融情感模型)
- ND4J

## 一、多股票新闻处理架构设计

### 1.1 核心挑战与解决方案

| 挑战类型 | 示例 | 问题 | 解决方案 |
|----------|------|------|----------|
| 多股票关联 | "茅台五粮液双双大涨" | 同一新闻涉及多只股票 | 句子分割+实体识别 |
| 情感不一致 | "宁德时代大涨，比亚迪大跌" | 同一新闻对不同股票情感相反 | 句子级情感+股票级聚合 |
| 行业新闻 | "新能源政策利好" | 影响板块，需分发到个股 | 行业映射+批量分发 |

### 1.2 处理流水线设计

```
原始新闻 → 篇章分析 → 句子分割 → 实体识别 → 句子级情感 → 股票级聚合 → 标签生成
```

**每个环节的设计要点**：

| 环节 | 核心任务 | 关键设计 |
|------|----------|----------|
| 句子分割 | 将新闻文本切分为独立语义单元 | 支持中文标点，保留上下文关联 |
| 实体识别 | 识别每只股票的提及位置 | 多策略匹配（代码/全称/简称/曾用名） |
| 情感分析 | 针对特定股票的句子级情感 | 上下文裁剪，排除其他股票干扰 |
| 结果聚合 | 综合多句情感得到股票级结果 | 加权平均（权重来自置信度、句子位置） |
| 标签生成 | 转换为训练标签 | 与股价涨幅阈值对齐 |

### 1.3 数据流向设计

```
新闻表
  ↓
新闻-句子表（一条新闻拆为多句）
  ↓
句子-股票关联表（一句可能关联多只股票）
  ↓
股票级情感表（每只股票的情感得分）
  ↓
训练样本表（股票代码 + 情感文本 + 涨跌标签）
```

## 二、股票实体识别方案

### 2.1 多策略匹配机制

| 策略 | 匹配对象 | 示例 | 置信度 | 适用场景 |
|------|----------|------|--------|----------|
| 精确代码 | 6位数字代码 | "600036" | 高(0.95) | 研报、公告 |
| 股票全称 | 完整公司名称 | "贵州茅台酒股份有限公司" | 高(0.9) | 正式文档 |
| 股票名称 | 股票简称 | "贵州茅台" | 中(0.85) | 多数新闻 |
| 市场简称 | 通用简称 | "茅台" | 中(0.8) | 口语化表达 |
| 曾用名 | 历史名称 | 查询公司曾用名 | 低(0.6) | 历史新闻 |

### 2.2 歧义处理策略

- **上下文消歧**: 分析句子中的行业关键词（"保险"指向中国平安，"银行"指向平安银行）
- **共现消歧**: 同一句子中是否出现股票代码
- **置信度惩罚**: 共享简称的多只股票降低匹配权重

### 2.3 特殊场景处理

#### 2.3.1 行业/板块新闻

当新闻讨论整个行业（如"新能源政策利好"）时：
- 不直接分配到具体股票，作为"行业信号"单独存储
- 可创建"行业样本"，与该行业所有股票对齐，赋予较低置信度

#### 2.3.2 新闻中未明确提及股票代码

如"刚刚，这家公司发布了重磅公告"：
- 维护"公司-股票映射表"，包含公司全称、常用简称
- 回溯匹配：如果新闻中无代码但有公司名称，映射到对应股票
- 间接匹配的置信度应低于直接代码匹配

#### 2.3.3 多条新闻的时间聚合

同一只股票在短时间（如30分钟）内有多个新闻：
- 新闻有时效性，设计衰减机制
- 最近新闻权重更高，考虑时间衰减因子
- 有新新闻出现时更新股票综合情感状态

## 三、句子级情感分析方案

### 3.1 上下文裁剪策略

针对每个句子-股票组合，需要提取该股票在句子中的局部上下文：

```
原句："今日茅台股价突破2000元，再创历史新高，带动白酒板块整体上涨，但五粮液因减持传闻小幅下跌。"

针对茅台提取："茅台股价突破2000元，再创历史新高"
针对五粮液提取："五粮液因减持传闻小幅下跌"
```

**裁剪原则**：
- 窗口大小：以股票名称为中心，前后各取一定字符数（如中文30个字符）
- 边界检测：以标点符号为自然边界，避免截断语义
- 保留关键信息：如数字（涨跌幅）、价格、关键词（减持、超预期等）

### 3.2 为什么需要句子级分析

直接对整篇新闻进行情感分析会导致严重的信息损失：

| 场景 | 整篇分析结果 | 句子级分析结果 |
|------|--------------|----------------|
| "茅台大涨，五粮液下跌" | 中性或混乱 | 茅台积极，五粮液消极 |
| "宁德时代业绩超预期，带动新能源板块" | 积极 | 宁德时代积极，其他相关股票需单独判断 |

### 3.3 情感分析模型要求

- 输入长度：支持短文本（50-200字符）
- 输出维度：三分类（积极/中性/消极）或回归得分（-1到1）
- 领域适应：理解"增持"、"下调"、"超预期"等金融术语

### 3.2 情感聚合权重设计

| 权重因子 | 设计考虑 | 示例 |
|----------|----------|------|
| 匹配置信度 | 实体识别越准确，权重越高 | 代码匹配(0.95) > 简称匹配(0.8) |
| 句子位置 | 标题中的句子权重更高 | 标题权重×1.5，正文×1.0 |
| 情感强度 | 情感越强烈，权重越高 | 积极/消极概率 > 0.8时权重增加 |
| 时效性 | 新闻开头的信息更重要 | 按句子顺序递减权重 |

**加权聚合公式**：

```
最终情感得分 = Σ(句子情感得分 × 权重) / Σ权重
```

### 3.3 情感标签映射

将情感得分映射为训练标签时，需要与股价涨幅阈值对齐：

| 情感得分范围 | 情感标签 | 股价标签 | 说明 |
|--------------|----------|----------|------|
| > 0.3 | 积极 | 涨(2) | 明显正面信息 |
| -0.3 到 0.3 | 中性 | 平(1) | 无明显倾向 |
| < -0.3 | 消极 | 跌(0) | 明显负面信息 |

**说明**：阈值（0.3）需要根据实际数据调整，也可设置为可配置参数。

## 四、情感模型选型

### 4.1 DJL生态内中文金融情感模型对比

| 方案 | 模型来源 | 中文金融适配 | 部署成本 | 整体评估 |
|------|----------|--------------|----------|----------|
| FinBERT系列 | HuggingFace PyTorch | ⭐⭐⭐⭐⭐ | 低 | **首选** |
| 轩辕2.0 | 度小满金融大模型 | ⭐⭐⭐⭐⭐ | 高 | 能力过剩，部署复杂 |
| FinGPT | 开源金融大模型 | ⭐⭐⭐⭐ | 中 | 学术友好，生产案例少 |
| ONNX通用模型 | 各类转换模型 | ⭐⭐ | 低 | 引擎中立，但模型少 |
| DJL Model Zoo | 内置预训练模型 | ⭐ | 最低 | 方便但无中文金融 |

### 4.2 FinBERT入选理由

| 维度 | 优势说明 |
|------|----------|
| 领域匹配 | 320亿+token中文金融语料预训练，情感分析F1达0.895 |
| 模型规模 | BERT-base(110M参数)，CPU可实时推理 |
| DJL支持 | PyTorch引擎成熟，TorchScript导出路径清晰 |
| 生态活跃 | HuggingFace多个变体，社区资源丰富 |

## 六、验证与评估设计

### 6.1 人工验证样本

随机抽取一定数量的新闻，人工标注每只股票的真实情感，评估：
- 实体识别准确率：是否准确识别出新闻涉及的所有股票
- 情感分析准确率：对每只股票的情感判断是否正确

### 6.2 与股价的关联验证

验证情感分析结果与股价变化的相关性：
- 一致性检查：积极情感是否对应上涨，消极情感是否对应下跌
- 相关性指标：计算情感得分与涨跌幅的相关系数
- 分组验证：按情感强度分组，检查各组平均涨幅是否合理

## 七、最终训练样本的生成

经过以上处理，每条新闻可能会生成多个训练样本：

| 新闻ID | 股票代码 | 情感文本 | 情感得分 | 股价标签 |
|--------|----------|----------|----------|----------|
| 1001 | 600036.SH | "茅台股价突破2000元，再创历史新高" | 0.85 | 2(涨) |
| 1001 | 000858.SZ | "五粮液因减持传闻小幅下跌" | -0.72 | 0(跌) |

这种设计确保了：
- ✅ 细粒度：每只股票都有独立的训练样本
- ✅ 准确性：文本只包含与该股票相关的上下文
- ✅ 可扩展：支持任意数量的股票和情感组合

## 九、数据库表设计

### 9.1 情感分析相关表

```sql
-- 新闻表
CREATE TABLE news (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    title VARCHAR(500),
    content TEXT,
    publish_time DATETIME,
    source VARCHAR(100),
    url VARCHAR(500),
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 新闻-句子表
CREATE TABLE news_sentence (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    news_id BIGINT,
    sentence_text TEXT,
    sentence_order INT,
    is_title BOOLEAN DEFAULT FALSE,
    FOREIGN KEY (news_id) REFERENCES news(id)
);

-- 句子-股票关联表
CREATE TABLE sentence_stock (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    sentence_id BIGINT,
    stock_code VARCHAR(10),
    stock_name VARCHAR(100),
    confidence DECIMAL(3,2),
    context_text TEXT,
    FOREIGN KEY (sentence_id) REFERENCES news_sentence(id)
);

-- 股票级情感表
CREATE TABLE stock_sentiment (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    stock_code VARCHAR(10),
    news_id BIGINT,
    sentiment_label VARCHAR(20),
    sentiment_score DECIMAL(5,4),
    confidence DECIMAL(5,4),
    analyzed_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (news_id) REFERENCES news(id)
);

-- 训练样本表
CREATE TABLE training_sample (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    stock_code VARCHAR(10),
    text_content TEXT,
    sentiment_label VARCHAR(20),
    sentiment_score DECIMAL(5,4),
    price_change DECIMAL(10,4),
    label_date DATE,
    is_labeled TINYINT DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_stock_code (stock_code),
    INDEX idx_labeled (is_labeled)
);

-- 公司-股票映射表
CREATE TABLE company_stock_mapping (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    stock_code VARCHAR(10) NOT NULL,
    stock_name VARCHAR(100) NOT NULL,
    company_full_name VARCHAR(200),
    industry VARCHAR(50),
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uk_code (stock_code)
);
```

### 5.1 数据源选型对比

| 方案 | 数据覆盖 | Java集成 | 免费性 | 中文 | 评估 |
|------|----------|----------|--------|------|------|

- **完全可控**: 数据存本地，不受第三方API限制
- **自动编码**: XmlReader自动处理中文乱码

### 6.1 Tushare日线方案

- **数据内容**: A股全部股票历史日线（2000年至今）
